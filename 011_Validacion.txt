VALIDAR LOS FORMULARIOS

En el Capítulo 5 vimos cómo programar nuestros objetos ActionForm para que se validen los datos ingresados por el usuario.
Este enfoque es útil para validaciones complejas; sin embargo, la mayoría de las veces las validaciones que necesitaremos realizar
en los formularios son muy simples: que un campo no quede vacío, que un campo sea numérico, que un campo contenga una fecha válida, etc.
En estos casos, implementar el método validate para repetir las mismas validaciones básicas es un proceso que podría simplificarse.
Además, se suma el hecho de que probablemente querremos usar forms como DynaActionForm o LazyValidatorForm para evitar crear una clase
por cada formulario, y en esos casos no tendremos dónde implementar el método validate.

Para todos estos casos, Struts Validator es la herramienta a usar.

El Struts Validator es un componente muy poderoso. Simple de usar para casos sencillos, sin embargo, provee funcionalidades como para 
poder definir cualquier tipo de validador propio.
Para empezar a usar el validador, primeramente 
debemos declararlo en el archivo: Validacion/WEB-INF/struts-config.xml
El validador es un plugin, esto es, un componente que Struts carga al iniciarse y destruye al finalizar su ejecución.

Los elementos plugin son el último tipo de elemento que definimos en el archivo de configuración: Validacion/WEB-INF/struts-config.xml
Veamos cómo definir el validador: 
"
<plug-in class-name="org.apache.struts.validator.ValidatorPlugin">
    <set-property property="pathnames" value="/WEB-INF/validations/validator-rules.xml"/>
    <set-property property="stopOnFirstError" value="true"/>
</plug-in>
"
Se crea el directorio: Validacion/WEB-INF/validations
Se crea el archivo: Validacion/WEB-INF/validations/validator-rules.xml

El elemento plug-in toma un atributo de nombre className que especifica la clase 
(que ha de implementar la interfaz org.apache.struts.action.Plugin) a instanciar.
Dentro del cuerpo del elemento podemos definir un conjunto de elementos set-property, 
que establecen valores a propiedades de la clase al instanciarse.

En el caso de la clase ValidatorPlugin, las propiedades a las que podemos establecer su valor son las siguientes:

pathnames: una lista de nombres de archivo, separados por coma, que contienen las descripciones de las validaciones a realizar.
Es muy importante no olvidar agregar los archivos xml que vayamos creando a esta lista, si no, por más que definamos el resto de 
las configuraciones bien, la validación no se realizará.

stopOnFirstError: si es true, el validador detendrá la validación JavaScript al primer error encontrado y mostrará dicho error. Caso contrario,
si es false, se validarán todos los campos y se mostrarán todos los errores juntos.

Como vemos, en principio estamos definiendo solamente un archivo de configuración de validación.
El archivo: Validacion/WEB-INF/validations/validator-rules.xml
(ubicado en la carpeta raíz de la distribución Library de Struts) 
contiene las descripciones de las validaciones básicas que provee Struts en base al proyecto Commons Validator
(URL: http://jakarta.apache.org/commons/validator). 
Estas validaciones estándar usan ciertas claves del archivo de recursos para mostrar los errores, por eso para su correcto uso 
es necesario que agreguemos al archivo de recursos las siguientes claves (los valores podemos cambiarlos a nuestro gusto o idioma):

Archivo de recursos: Validacion/WEB-INF/MessageResources.properties 
"
errors.required={0} es requerido.
errors.minlength={0} no puede ser menor de {1} caracteres.
errors.maxlength={0} no puede ser mayor de {1} caracteres.
errors.invalid={0} es inválido.
errors.byte={0} debe ser un byte.
errors.short={0} debe ser un short.
errors.integer={0} debe ser un integer.
errors.long={0} debe ser un long.
errors.float={0} debe ser un float.
errors.double={0} debe ser un double.
errors.date={0} no esuna fecha.
errors.range={0} no está en el rango {1} - {2}.
errors.creditCard={0} es una tarjeta de crédito inválida.
errors.email={0} no es una dirección de correo electrónico válida.
errors.url={0} no es una dirección web válida.
"

En cuanto conozcamos mejor cada tipo de validación veremos cómo pueden ser utilizados estos mensajes.

Definir un validador
Para definir validaciones sobre un formulario, debemos proveerle al validador un archivo xml. Técnicamente, 
podríamos agregar al archivo: Validacion/WEB-INF/validations/validator-rules.xml 
los elementos XML para las validaciones, pero es una práctica no recomendada.
Una buena práctica es crear un archivo XML para cada formulario que deseemos validar.
Veamos en qué consiste este archivo: Validacion/WEB-INF/validations/validator-rules.xml
"
<!DOCTYPE form-validation PUBLIC
    "-//Apache Struts//XWork Validator 1.0.2//EN"
    "http://struts.apache.org/dtds/xwork-validator-1.0.2.dtd">

<!-- Antes se hacía así para la versión 1 de Struts  
<!DOCTYPE form-validation PUBLIC
    "-//Apache Software Foundation//DTD Commons Validator Rules Configuration 1.1.3//EN"
    "http://jakarta.apache.org/commons/dtds/validator-1.1.3.dtd">
-->

<!-- Elemento raíz -->
<form-validation>

<!-- Conjunto de formularios -->
    <formset>
    <!-- Agregamos nuestro formulario al conjunto -->
        <form name="proyectoForm">

            <!-- Primer Campo a validar -->
            <field property="nombre" depends="required">
                <arg position="0" key="proyecto.nombre"/>
            </field>

            <!-- Segundo Campo a validar -->
            <field property="fecha" depends="date">
                <arg position="0" key="proyecto.fecha"/>
                <var>
                    <var-name>datePattern</var-name>
                    <var-value>MM-dd-yyyy</var-value>
                </var>
            </field>
        </form>
    </formset>
</form-validation>

"

El validador internamente tiene un conjunto de objetos llamados formset, 
cada elemento form dentro de este elemento agrega un formulario
al conjunto de formularios que tendrán validación.

Veamos elemento por elemento el archivo: Validacion/WEB-INF/validations/validator-rules.xml

<form name="proyectoForm">
Mediante este elemento declaramos que estamos definiendo la validación para el formulario de nombre proyectoForm.
Este form tiene que existir con este nombre en el archivo: Validacion/WEB-INF/struts-config.xml.

<field property="nombre" depends="required">
    <arg position="0" key="proyecto.nombre"/>
</field>
En un formulario, podemos validar uno o más campos. Cada elemento field define un campo y las validaciones que han de 
realizarse sobre él cuando el formulario sea enviado. El campo que validamos lo definimos con el atributo property. 
Las validaciones que se efectuarán sobre el campo las definimos mediante el atributo depends. Si queremos realizar más
de una validación, las escribimos separadas por comas. Un campo debe pasar todas las validaciones para que sea válido. 
En el cuerpo del tag field insertaremos elementos y propiedades diversas, dependiendo del tipo de validación que estemos
efectuando.

Validaciones estándar
Vamos a ver las validaciones que Struts provee, su uso y cómo configurarlas. En todos los casos, los elementos arg dentro del
elemento field definen la clave del archivo de recursos con la etiqueta del campo a ser usada por el mensaje de error.
Por ejemplo, para el validador required el mensaje de error es:

errors.required={0} es requerido.

Si nuestro arg de posición 0 es, como en el ejemplo, proyecto.nombre y su valor en castellano es nombre, 
entonces al producirse un error que es mostrado en la página veremos lo siguiente:

nombre es requerido

required
Valida que el campo contenga un valor. Ejemplo de uso:

<field property="nombre" depends="required">
    <arg position="0" key="proyecto.nombre" />
</field>

minlength, maxlength
Valida que el campo tenga un número mínimo o máximo de caracteres, respectivamente. 
Este elemento toma una variable especificando el valor de mínimo o máximo, siendo minlength y maxlength.
Ejemplo de uso:

<field property="nombre" depends="minlength">
    <arg position="0" key="form.nombre" />
    <arg position="1" key="5">
    <var>
        <var-name>minlength</var-name>
        <var-value>5</var-value>
    </var>
</field>

Este es un ejemplo para validar un campo con un número mínimo de caracteres.
Para validar por el máximo, el procedimiento es igual, pero reemplazando por maxlength en vez de minlength.

Viendo este ejemplo notamos dos inconvenientes. El primero es que tenemos que repetir el valor mínimo (o máximo)
tanto en la variable como en el argumento del mensaje. El segundo, y más importante, es que el segundo atributo de arg 
es key, o sea, la clave del una entrada en el archivo de recursos por cada número que usemos en los validadores!

Afortunadamente, podemos resolver estos dos inconvenientes. Para el primer problema, simplemente podemos establecer 
una referencia al valor de la variable que definimos en el elemento var. Para el segundo problema, existe el atributo
resource dentro de los elementos arg que especifica si el valor es una clave del archivo de recursos o es un valor literal.
Veamos entonces el mismo ejemplo anterior pero con estas salvedades.

<field property="nombre" depends="minlength">
    <arg position="0" key="form.nombre" />
    <arg position="1" name="minlength" key="${var:minlength}" resource="false" />
    <var>
        <var-name>minlength</var-name>
        <var-value>5</var-value>
    </var>
</field>

El atributo key del segundo elemento arg hace referencia al valor de la variable minlength, y a su vez define
que dicho valor no es una clave del archivo de recursos sinó el valor en sí.

mask
Valida el campo según una expresión regular. Si el valor del campo se ajusta a la expresión regular, entonces se lo 
considera válido. Este validador es muy útil para validar formatos específicos en los campos. Veamos un ejemplo. 
Supongamos que un campo del formulario pide al usuario el código postal de su residencia. Supongamos también que todo
código postal válido consta de 3 letras y 4 dígitos. Entonces definimos el siguiente validador:

<field property="cp" depends="mask" >
    <msg name="mask" key="form.cp.msg" />
    <arg key="form.cp" position="0" />
    <var>
        <var-name>mask</var-name>
        <var-value>^[a-zA-Z]{3}[0-9]{4}$</var-value>
    </var>
</field>

Este validador se encargará de validar la propiedad cp del formulario. En la variable mask definimos la expresión regular. 
Ésta debe empezar con el signo ^ y debe terminar con el signo $.

El validador mask es un validador genérico y, por consiguiente, no tiene un mensaje de error por defecto.
En el ejemplo vemos el elemento msg. Dicho atributo especifica la clave, para el validador mask, del mensaje de error.
Si el mensaje de error tuviera variables, además podemos especificar un elemento arg.

Las entradas en el archivo de recursos correspondientes son:

form.cp=Codigo postal
form.cp.msg={0} debe ser de la forma ABC1234

byte, short, integer, long, float, double
Estos validadores verifican que el valor del campo pueda ser convertido al tipo dado. 
En todos los casos no se necesita otra variable más que el argumento para el mensaje de error.
Veamos dos ejemplos:

<field property="edad" depends="integer" >
    <arg key="form.edad" position="0" />
</field>

<field property="sueldo" depends="double" >
    <arg key="form.sueldo" position="0" />
</field>

date
Verifica que el valor del campo represente una fecha, según el patrón de conversión dado. Las posibles variables
son datePattern y datePatternStrict. La diferencia entre estos dos es que datePatternStrict, como su nombre lo indica, es 
estricto en cuanto a la cantidad de caracteres. Por ejemplo, con un patrón dd/MM/yyyy la fecha 4/6/1979 fallará si el patrón es
declarado como datePatternStrict. Para este caso la fecha correcta sería 04/06/1979.

<field property="fecha_nacimiento" depends="date" >
    <arg key="form.fecha_nacimiento" position="0" />
    <var>
        <var-name>datePattern</var-name>
        <var-value>dd/MM/yyyy</var-value>
    </var>
</field>

intRange, floatRange, doubleRange
