DESENTRAÑAR EL STRUTS-CONFIG 
El archivo Configuracion/WEB-INF/struts-config.xml es la base de la configuración de Struts.
En los capítulos previos hemos visto cómo usarlo para configurar nuestras acciones, formularios, validaciones, etc. 
Además de todo eso, este archivo posee muchas otras variantes de configuraciones para ajustar su funcionamiento a nuestras necesidades.

Al ser el struts-config un archivo XML, no sólo su contenido debe ser XML bien formado, sinó que además debe respetar un orden. 
Sus posibles elementos son (manteniendo el orden):

display-name?, description?, data-sources?, form-beans?, global-exceptions?, global-forwards?, action-mappings?, controller?, message-resources*, plug-in* 

El signo ? representa un elemento opcional y 
el signo * representa que puede haber cero o más elementos.

display-name
Este elemento se emplea para darle un nombre a la aplicación. Veremos que se utiliza en varias partes del archivo, generalmente por aplicaciones gráficas.

description
Al igual que display-name, provee una descripción de la aplicación. Estos dos elementos no tienen inferencia en la funcionalidad de la aplicación.

data-sources
Este elemento engloba cero o más elementos data-source. Cada elemento de este tipo define una conexión a una base de datos. Definir las
conexiones en estos elementos permite a las acciones obtener fácilmente conexiones sin necesidad de tener que programar el código para 
obtenerlas en cada acción.
Una práctica recomendable es usar el patrón DAO: Data Access Object. Este patrón, resumidamente, indica que cada objeto es responsable de 
su propia persistencia. De esta forma, las acciones interactúan con objetos que, cuando corresponde, se guardan a sí mismos en la base de datos,
se actualizan, etc.

El elemento data-source define muy pocos atributos. La configuración propia de la conexión se define mediante un conjunto de 
atributos set-property del cuerpo del elemento.

key: clave bajo la cual será guardado el objeto.

type: nombre de la clase Java (debe implementar la interfaz javax.sql.DataSource) del datasource a definir.

Veamos un ejemplo. Definimos una conexión a la base usando el proyecto Apache Commons DBCP. La clase org.apache.commons.dbcp.BasicDataSource implementa la interfaz javax.sql.Datasource. Es un Java Bean con muchas propiedades que debemos definir con elementos set-property para 
poder establecer una conexión:
Archivo: Configuracion/WEB-INF/struts-config.xml 
"
<?xml version="1.0" encoding="ISO-8859-1" ?>

<!DOCTYPE struts-config PUBLIC
	"-//Apache Software Foundation//DTD Struts Configuration 1.2//EN"
	"http://jakarta.apache.org/struts/dtds/struts-config_1_2.dtd">
	
<struts-config>

    <!-- Elemento display-name-->
    <display-name>Ejemplo de validación con Struts</display-name>

    <!-- Elemento description -->
    <description>Ejemplo de validación con Struts</description>

    <!-- Elemento data-sources que contiene elementos data-source que define una conexión a una base de datos -->
    <data-sources>
        <data-source type="org.apache.commons.dbcp.BasicDataSource">
            <set-property property="driverClassName" value="com.mysql.cj.jdbc.Driver"/>
            <set-property property="url" value="jdbc:mysql://localhost:3306/miBaseDeDatos"/>
            <set-property property="username" value="andres"/>
            <set-property property="password" value="malacate"/>
            <set-property property="maxActive" value="10"/>
            <set-property property="maxWait" value="5000"/>
            <set-property property="defaultAutoCommit" value="false"/>
            <set-property property="validationQuery" value="SELECT COUNT(*) FROM tutoriales"/>
        </data-source>
    </data-sources>

    

	<action-mappings>
		<!-- Acción 1-->
		<action
			path="/cliente" name="clienteForm" scope="request" validate="false"
            forward="/jsp/cliente.jsp" />	
		</action>
        <!-- Acción 2-->
		<action
			path="/cliente2"
            type="action.ProcesarClienteAction" 
            name="clienteForm" scope="request" validate="true"
            input="/cliente.do" >	
		</action>

		<!-- Ejemplo para la validación multi-página -->
		<!-- Definimos tres acciones -->
		<!-- Muestra la primera página -->
		<action 
			path="/wizard1" name="wizardForm" validate="false"
			forward="/jsp/wizard1.jsp" />
		
		<!-- Valida la primera página y muestra la segunda página -->
		<action 
			path="/wizard2" name="wizardForm" scope="request" validate="true"
			input="/jsp/wizard1.jsp" 
			forward="/jsp/wizard2.jsp" />
		
		<!-- Valida la primera y segunda página y ejecuta la acción si la validación fue exitosa -->
		<action
			path="/procesarWizardForm"
			type="action.ProcesarWizardAction"
			name="wizardForm" scope="request" validate="true"
			input="/jsp/wizard2.jsp" />
		</action>

	</action-mappings>

    <plug-in class-name="org.apache.struts.validator.ValidatorPlugin">
        <set-property property="pathnames" value="/WEB-INF/validations/validator-rules.xml"/>
        <set-property property="stopOnFirstError" value="true"/>
    </plug-in>

</struts-config>
"

Recordemos que para que el ejemplo funcione debemos descargar la librería del proyecto Apache Commons DBCP 
y agregarla a la aplicación en la carpeta Configuracion/WEB-INF/lib.
Nota: URL de Apache Commons DBCP: https://commons.apache.org/dbcp/download_dbcp.cgi
Archivo descargado: https://dlcdn.apache.org//commons/dbcp/binaries/commons-dbcp2-2.14.0-bin.zip
Se descomprime en una carpeta de descargas y se copian los archivos *.jar :
- commons-dbcp2-2.14.0.jar 
- commons-dbcp2-2.14.0-javadoc.jar
- commons-dbcp2-2.14.0-sources.jar
- commons-dbcp2-2.14.0-test.jar
- commons-dbcp2-2.14.0-test-sources.jar
dentro de la carpeta: Configuracion/WEB-INF/lib.
Luego se ejecuta el Eclipse IDE para agregar estas librerías de Apache Commons DBCP al Java Build Path (Classpath)
(Properties -> Java Buidl Path -> [Add external jar])  se aplica y se cierra Eclipse IDE.

Luego desde una acción podremos obtener la conexión de la siguiente forma: 
Su implementación en el Archivo: Configuracion/src/BDAction.java 
"
package action;

import java.sql.Connection;
import java.sql.SQLException;
import java.sql.Statement;

import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import javax.sql.DataSource;

import org.apache.struts.action.Action;
import org.apache.struts.action.ActionForm;
import org.apache.struts.action.ActionForward;
import org.apache.struts.action.ActionMapping;

public class BDAction extends Action 
{
    public ActionForward execute(ActionMapping map, ActionForm form, HttpServletRequest req, HttpServletResponse res) throws Exception
    {
        DataSource dataSource;
        Connection conexion = null;
        try
        {
            // Obtenemos el datasource por defecto
            dataSource = getDataSource(req);

            // Obtenemos la conexión del datasource
            conexion = dataSource.getConnection();

            // Código JDBC
            Statement stmt = conexion.createStatement();
            stmt.execute("SELECT * FROM PAIS");
        }
        // Debemos cerrar siempre la conexión
        finally
        {
            try
            {
                conexion.close();
            }
            catch(SQLException e)
            {
                // Registramos la excepción
            }
        }
        return map.findforward("ok");
    }
}
"

Si queremos definir más de un datasource, por ejemplo, para acceder a diferentes bases de datos, 
lo hacemos especificando el atributo key en el elemento data-source 
(dentro del archivo: Configuracion/WEB-INF/struts-config.xml).

<data-sources>

	<data-source key="epsilon" type="org.apache.commons.dbcp.BasicDataSource">
		<!-- Propiedades de la conexión -->
	</data-source>

	<data-source key="gama" type="org.apache.commons.dbcp.BasicDataSource">
		<!-- Propiedades de la conexión -->
	</data-source>

</data-sources>

Luego en la implementación de las acciones dentro de la clase Java BDAction.java las referenciamos usando 
la clave en el método getDataSource:
"
...
DataSource epsilon = getDataSource(req, "epsilon");
...
DataSource gama = getDataSource(req, "gama");
"

form-beans
Elemento que define los forms (elemento form-bean) que se usarán en la aplicación.
Lo vimos con profundidad en el Capítulo 5.

global-exceptions
Elemento raíz de un conjunto de excepciones que se definen para ser manejadas en caso de que una acción
las arroje. Cada elemento exception perteneciente a este elemento puede contener estos atributos:

handler: nombre de la clase Java que manejará la excepción. 
Por defecto es org.apache.struts.action.ExceptionHandler.

key: clave del archivo de recursos (archivo: Configuracion/WEB-INF/MessageResources.properties) 
que contiene el mensaje de error a mostrar. 
El mensaje de la excepción será pasado como argumento.

path: dirección adonde se redirigirá la acción cuando ocurra esta excepción.

type: nombre de la clase Java de la excepción.

En el elemento global-exceptions estamos definiendo excepciones globales, cada acción puede definir también
su excepción local, que sobreescribirá la global eventualmente. 
Veamos un ejemplo en donde definimos una excepción global y una local:
(dentro del archivo. Configuracion/WEB-INF/struts-config.xml)
"
<!-- Elemento global-exceptions que contiene elementos exception que definen cómo manejar las excepciones en la aplicación -->
	<!-- La excepción global -->
	<global-exceptions>
		<exception
			key="excepcion.null"
			path="/jsp/nullException.jsp"
			type="java.lang.NullPointerException"
			/>
	</global-exceptions>

	<!-- Acción con Excepción local -->
	<action 
		path="/accionXYZ" 
		scope="request"
		type="XYZAction"

		<exception
			key="excepcion.formato" 
			path="/jsp/nfException.jsp" 
			type="java.lang.NumberFormatException" />
		/>
		<forward 
			name="ok" 
			path="/jsp/xyz.jsp"/>
	</action>
"

La excepción global hace que cada acción que arroje una excepción del tipo NullPointerException 
no capturada se intercepte, se creee un mensaje de error con la clave dada y se redirija a la página.
Lo mismo ocurre con las excepciones de tipo NumberFormatException, pero solamente cuando ocurren durante la ejecución de la acción XYZAction.

global-forwards
Al igual que con las excepciones, podemos definir forwards globales, además de los locales de cada acción.
Esto es útil para definir un forward global de error, por ejemplo, y saber que todas las acciones podrán hacer
uso de él sin necesidad de tener que definirlo en cada una de las acciones.
Veamos un ejemplo (dentro del archivo: Configuracion/WEB-INF/struts-config.xml)
"
<!-- Ejemplo de Elemento global-forwards -->
	<global-forwards>
		<forward 
			name="mal" 
			path="/jsp/form_mal.jsp" />
	</global-forwards>
"

action-mappings
En este elemento definimos las acciones de la aplicación. Si bien hemos visto varios ejemplos en lo que 
va del libro, vamos a ver ahora una posible configuración muy útil. Al definir los action-mappings podemos
usar comodines para combinar mapeos similares y reducir el número de elementos action-mapping.
Veamos primero un ejemplo y luego la explicación (dentro del archivo: Configuracion/WEB-INF/struts-config.xml):
"
<!-- Ejemplo de uso de comodines en la definición de acciones -->
		<action
			path="/ver*" 
			type="paquete.Ver{1}Action" 
			name="{1}Form" 
			scope="request" 
			validate="false" >
			<forward name="ok" path="/jsp/{1}.jsp" />
		</action>
"

En este caso, esta meta acción se define con el comodín * que permite manejar direcciones como verSaldo, verCuentas, etc. El contenido que se asocie al comodín será reemplazado luego en las variables {1}, {2}, etc.
Como ejemplo, si ingresamos a la dirección URL: http://localhost:8080/Struts/verSaldo.do esta acción se comportará como si hubiese sido definida de la siguiente forma:
(dentro del archivo: Configuracion/WEB-INF/struts-config.xml)
"
<action
	path="/verSaldo"
	type="paquete.VerSaldoAction"
	name="SaldoForm"
	scope="request"
	validate="false" >
	<forward name="ok" path="/jsp/Saldo.jsp" />
</action>
"

Otro ejemplo más complejo, incluyendo varios comodines:
(dentro del archivo: Configuracion/WEB-INF/struts-config.xml)
"
<!-- Otro ejemplo más complejo, incluyendo varios comodines en la definición de acciones -->
		<action
			path="/*/mostrar*" 
			type="{1}.Mostrar{2}Action" 
			name="{2}Form" 
			scope="request" 
			validate="false" >
			<forward name="ok" path="/jsp/{1}/{2}.jsp" />
		</action>
"

En este ejemplo, el contenido que aparece al primer * reemplazará luego a la variable {1}
y el contenido que aparece al segundo * reemplazará a {2}. Si ingresamos en la URL 
http://localhost:8080/Struts/finanzas/mostrarBolsa.do esta acción se comportará como si hubiese sido definida:
(dentro del archivo: Configuracion/WEB-INF/struts-config.xml)
"
<action
	path="/finanzas/mostrarBolsa"
	type="finanzas.MostrarBolsaAction"
	name="BolsaForm"
	scope="request"
	validate="false" >
	<forward name="ok" path="/jsp/finanzas/Bolsa.jsp" />
</action>
"

Recordemos que la porción /Struts de la dirección URL se refiere al contexto dentro del Apache Tomcat,
y no es tomada en cuenta por Struts.

Notemos en este segundo ejemplo que el comodín * no aparea cadenas de texto que incluyan el caracter /.
Si queremos que incluya también los caracteres /, debemos usar el comodín **.

Los atributos type, name, roles, parameter, attribute, forward, include e input del elemento action aceptan comodines.
El atributo path del elemento forward también acepta comodines.

controller
Con este elemento podemos configurar a medida el controlador, o sea, el ActionServlet que recibe los pedidos
y se encarga de manejarlos y redirigirlos a las acciones correspondientes. Muchos atributos de la configuración se refieren al envío de archivos de parte del usuario, veámoslos:

bufferSize: tamaño del buffer de entrada que maneja la subida de archivos.

maxFileSize: tamaño máximo aceptable para los archivos que se suban. Expresado en bytes, si le agregamos el 
sufijo K, M o G expresa kilobytes, megabytes o gigabytes respectivamente. Por defecto: 250M.

memFileSize: tamaño máximo que será guardado en memoria cuando se está subiendo un archivo. 
Una vez alcanzado este valor, se procederá a guardar el contenido (por lo general en el disco). 
Expresado en bytes, también podemos agregar el sufijo K, M o G. Por defecto: 256K.

tempDir: carpeta temporaria donde se guardará el archivo que se está subiendo.

multipartClass: nombre de la clase que manejará los pedidos que incluyan archivos (pedidos multipart).
Por defecto: org.apache.struts.upload.CommonsMultipartRequestHandler.

contentType: tipo de contenido MIME (opcionalmente también el mapa de caracteres) que se devolverá
por defecto en cada respuesta. Este valor puede ser redefinido particularmente por cada acción o página JSP.
Por defecto, text/html.

forwardPattern: patrón que especifica cómo transformar el valor del atributo path del elemento forward 
a una dirección web (relativa al contexto). Este patrón puede contener los siguientes valores:
	$M el nombre del módulo actual 
	$P el atributo path del elemento forward
	$$ se traduce al signo $, usado como secuencia de escape.
Este atributo rara vez necesita ser modificado. Por defecto, $M$P.

pagePattern: patrón que especifica cómo transformar el valor del atributo page de los tags que lo usen 
(por ejemplo, html:link). Su formato es el mismo que en forwardPattern. Por defecto: $M$P.

inputForward: si es true, el valor del atributo input de los elementos action será tomado como referencia 
a un elemento forward, local o global. Si es false (por defecto), dicho valor será tomado como una dirección web relativa al módulo.
Veamos el siguiente ejemplo:
(dentro del archivo: Configuracion/WEB-INF/struts-config.xml)
"
<!-- Ejemplo de acción que muestra el atributo inputForward -->
		<action 
			path="/topoto"
			type="paquete.TopotoAction"
			scope="request"
			validate="true"
			input="error">
			<forward name="ok" path="/jsp/topoto.jsp" />
			<forward name="error" path="/jsp/error.jsp" />
		</action>
"
Si configuramos el elemento inputForward con valor true, al ocurrir un error de validación, 
el atributo input está indicando el nombre de un elemento forward. En este caso, lo encontrará localmente
y entonces redirigirá la acción a la página /jsp/error.jsp.

locale: si es true (por defecto lo es), se guardará en la sesión del usuario un objeto de tipo Locale.

nocache: si es true, se agregará en cada respuesta encabezados para impedir que el navegador del usuario guarde
páginas en su caché. Por defecto es false.

processorClass: nombre de la clase que realiza la lógica de procesamiento de cada pedido. Esta clase es 
muy importante y luego veremos ejemplos de cómo extender la clase que brinda Struts para agregar funcionalidad.
Por defecto es: org.apache.struts.action.RequestProcessor.


message-resources
Este elemento lo usamos para agregar a nuestra aplicación archivos de recursos (como objetos de tipo MessageResources). Podemos definir cualquier número de message-resources.
Veamos sus atributos:

factory: nombre de la subclase de org.apache.struts.util.MessageResourcesFactory que usaremos como fábrica
del objeto MessageResources. Por defecto: org.apache.struts.util.PropertyMessageResourcesFactory, que asume 
nuestro archivo de recursos es un archivo de propiedades (o sea, de la forma clave=valor).

key: nombre del atributo donde el objeto será guardado en el ServletContext. Por defecto, la constante definida
por org.apache.struts.Globals.MESSAGES_KEY

null: si es true (por defecto lo es), al buscar una clave en el archivo de recursos que no exista,
nos devolverá null. Si no, al buscar una clave inexistente, obtendremos un error.

parameter: parámetro que pasaremos al objeto fábrica para crear el objeto. En el caso de PropertyMessageResourcesFactory, el nombre del archivo de recursos (pero sin la extensión ".properties").


plug-in
