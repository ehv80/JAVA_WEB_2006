DESENTRAÑAR EL STRUTS-CONFIG 
El archivo Configuracion/WEB-INF/struts-config.xml es la base de la configuración de Struts.
En los capítulos previos hemos visto cómo usarlo para configurar nuestras acciones, formularios, validaciones, etc. 
Además de todo eso, este archivo posee muchas otras variantes de configuraciones para ajustar su funcionamiento a nuestras necesidades.

Al ser el struts-config un archivo XML, no sólo su contenido debe ser XML bien formado, sinó que además debe respetar un orden. 
Sus posibles elementos son (manteniendo el orden):

display-name?, description?, data-sources?, form-beans?, global-exceptions?, global-forwards?, action-mappings?, controller?, message-resources*, plug-in* 

El signo ? representa un elemento opcional y 
el signo * representa que puede haber cero o más elementos.

display-name
Este elemento se emplea para darle un nombre a la aplicación. Veremos que se utiliza en varias partes del archivo, generalmente por aplicaciones gráficas.

description
Al igual que display-name, provee una descripción de la aplicación. Estos dos elementos no tienen inferencia en la funcionalidad de la aplicación.

data-sources
Este elemento engloba cero o más elementos data-source. Cada elemento de este tipo define una conexión a una base de datos. Definir las
conexiones en estos elementos permite a las acciones obtener fácilmente conexiones sin necesidad de tener que programar el código para 
obtenerlas en cada acción.
Una práctica recomendable es usar el patrón DAO: Data Access Object. Este patrón, resumidamente, indica que cada objeto es responsable de 
su propia persistencia. De esta forma, las acciones interactúan con objetos que, cuando corresponde, se guardan a sí mismos en la base de datos,
se actualizan, etc.

El elemento data-source define muy pocos atributos. La configuración propia de la conexión se define mediante un conjunto de 
atributos set-property del cuerpo del elemento.

key: clave bajo la cual será guardado el objeto.

type: nombre de la clase Java (debe implementar la interfaz javax.sql.DataSource) del datasource a definir.

Veamos un ejemplo. Definimos una conexión a la base usando el proyecto Apache Commons DBCP. La clase org.apache.commons.dbcp.BasicDataSource implementa la interfaz javax.sql.Datasource. Es un Java Bean con muchas propiedades que debemos definir con elementos set-property para 
poder establecer una conexión:
Archivo: Configuracion/WEB-INF/struts-config.xml 
"
<?xml version="1.0" encoding="ISO-8859-1" ?>

<!DOCTYPE struts-config PUBLIC
	"-//Apache Software Foundation//DTD Struts Configuration 1.2//EN"
	"http://jakarta.apache.org/struts/dtds/struts-config_1_2.dtd">
	
<struts-config>

    <!-- Elemento display-name-->
    <display-name>Ejemplo de validación con Struts</display-name>

    <!-- Elemento description -->
    <description>Ejemplo de validación con Struts</description>

    <!-- Elemento data-sources que contiene elementos data-source que define una conexión a una base de datos -->
    <data-sources>
        <data-source type="org.apache.commons.dbcp.BasicDataSource">
            <set-property property="driverClassName" value="com.mysql.cj.jdbc.Driver"/>
            <set-property property="url" value="jdbc:mysql://localhost:3306/miBaseDeDatos"/>
            <set-property property="username" value="andres"/>
            <set-property property="password" value="malacate"/>
            <set-property property="maxActive" value="10"/>
            <set-property property="maxWait" value="5000"/>
            <set-property property="defaultAutoCommit" value="false"/>
            <set-property property="validationQuery" value="SELECT COUNT(*) FROM tutoriales"/>
        </data-source>
    </data-sources>

    

	<action-mappings>
		<!-- Acción 1-->
		<action
			path="/cliente" name="clienteForm" scope="request" validate="false"
            forward="/jsp/cliente.jsp" />	
		</action>
        <!-- Acción 2-->
		<action
			path="/cliente2"
            type="action.ProcesarClienteAction" 
            name="clienteForm" scope="request" validate="true"
            input="/cliente.do" >	
		</action>

		<!-- Ejemplo para la validación multi-página -->
		<!-- Definimos tres acciones -->
		<!-- Muestra la primera página -->
		<action 
			path="/wizard1" name="wizardForm" validate="false"
			forward="/jsp/wizard1.jsp" />
		
		<!-- Valida la primera página y muestra la segunda página -->
		<action 
			path="/wizard2" name="wizardForm" scope="request" validate="true"
			input="/jsp/wizard1.jsp" 
			forward="/jsp/wizard2.jsp" />
		
		<!-- Valida la primera y segunda página y ejecuta la acción si la validación fue exitosa -->
		<action
			path="/procesarWizardForm"
			type="action.ProcesarWizardAction"
			name="wizardForm" scope="request" validate="true"
			input="/jsp/wizard2.jsp" />
		</action>

	</action-mappings>

    <plug-in class-name="org.apache.struts.validator.ValidatorPlugin">
        <set-property property="pathnames" value="/WEB-INF/validations/validator-rules.xml"/>
        <set-property property="stopOnFirstError" value="true"/>
    </plug-in>

</struts-config>
"

y agregarla a la aplicación en la carpeta Configuracion/WEB-INF/lib.
Nota: URL de Apache Commons DBCP: https://commons.apache.org/dbcp/download_dbcp.cgi
Archivo descargado: https://dlcdn.apache.org//commons/dbcp/binaries/commons-dbcp2-2.14.0-bin.zip
Se descomprime en una carpeta de descargas y se copian los archivos *.jar :

- commons-dbcp2-2.14.0-javadoc.jar
- commons-dbcp2-2.14.0-sources.jar
- commons-dbcp2-2.14.0-test.jar
- commons-dbcp2-2.14.0-test-sources.jar
dentro de la carpeta: Configuracion/WEB-INF/lib.
Luego se ejecuta el Eclipse IDE para agregar estas librerías de Apache Commons DBCP al Java Build Path (Classpath)
(Properties -> Java Buidl Path -> [Add external jar])  se aplica y se cierra Eclipse IDE.

Luego desde una acción podremos obtener la conexión de la siguiente forma: 
Su implementación en el Archivo: Configuracion/src/BDAction.java 
"
package action;

import java.sql.Connection;
import java.sql.SQLException;
import java.sql.Statement;

import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import javax.sql.DataSource;

import org.apache.struts.action.Action;
import org.apache.struts.action.ActionForm;
import org.apache.struts.action.ActionForward;
import org.apache.struts.action.ActionMapping;

public class BDAction extends Action 
{
    public ActionForward execute(ActionMapping map, ActionForm form, HttpServletRequest req, HttpServletResponse res) throws Exception
    {
        DataSource dataSource;
        Connection conexion = null;
        try
        {
            // Obtenemos el datasource por defecto
            dataSource = getDataSource(req);

            // Obtenemos la conexión del datasource
            conexion = dataSource.getConnection();

            // Código JDBC
            Statement stmt = conexion.createStatement();
            stmt.execute("SELECT * FROM PAIS");
        }
        // Debemos cerrar siempre la conexión
        finally
        {
            try
            {
                conexion.close();
            }
            catch(SQLException e)
            {
                // Registramos la excepción
            }
        }
        return map.findforward("ok");
    }
}
"

Si queremos definir más de un datasource, por ejemplo, para acceder a diferentes bases de datos, 
lo hacemos especificando el atributo key en el elemento data-source 
(dentro del archivo: Configuracion/WEB-INF/struts-config.xml).

<data-sources>

	<data-source key="epsilon" type="org.apache.commons.dbcp.BasicDataSource">
		<!-- Propiedades de la conexión -->
	</data-source>

	<data-source key="gama" type="org.apache.commons.dbcp.BasicDataSource">
		<!-- Propiedades de la conexión -->
	</data-source>

</data-sources>

Luego en la implementación de las acciones dentro de la clase Java BDAction.java las referenciamos usando 
la clave en el método getDataSource:
"
...
DataSource epsilon = getDataSource(req, "epsilon");
...
DataSource gama = getDataSource(req, "gama");
"

form-beans
Elemento que define los forms (elemento form-bean) que se usarán en la aplicación.
Lo vimos con profundidad en el Capítulo 5.

global-exceptions
Elemento raíz de un conjunto de excepciones que se definen para ser manejadas en caso de que una acción
las arroje. Cada elemento exception perteneciente a este elemento puede contener estos atributos:

handler: nombre de la clase Java que manejará la excepción. 
Por defecto es org.apache.struts.action.ExceptionHandler.

key: clave del archivo de recursos (archivo: Configuracion/WEB-INF/MessageResources.properties) 
que contiene el mensaje de error a mostrar. 
El mensaje de la excepción será pasado como argumento.

path: dirección adonde se redirigirá la acción cuando ocurra esta excepción.

type: nombre de la clase Java de la excepción.

En el elemento global-exceptions estamos definiendo excepciones globales, cada acción puede definir también
su excepción local, que sobreescribirá la global eventualmente. 
Veamos un ejemplo en donde definimos una excepción global y una local:
(dentro del archivo. Configuracion/WEB-INF/struts-config.xml)
"
<!-- Elemento global-exceptions que contiene elementos exception que definen cómo manejar las excepciones en la aplicación -->
	<!-- La excepción global -->
	<global-exceptions>
		<exception
			key="excepcion.null"
			path="/jsp/nullException.jsp"
			type="java.lang.NullPointerException"
			/>
	</global-exceptions>

	<!-- Acción con Excepción local -->
	<action 
		path="/accionXYZ" 
		scope="request"
		type="XYZAction"

		<exception
			key="excepcion.formato" 
			path="/jsp/nfException.jsp" 
			type="java.lang.NumberFormatException" />
		/>
		<forward 
			name="ok" 
			path="/jsp/xyz.jsp"/>
	</action>
"

La excepción global hace que cada acción que arroje una excepción del tipo NullPointerException 
no capturada se intercepte, se creee un mensaje de error con la clave dada y se redirija a la página.
Lo mismo ocurre con las excepciones de tipo NumberFormatException, pero solamente cuando ocurren durante la ejecución de la acción XYZAction.

global-forwards
Al igual que con las excepciones, podemos definir forwards globales, además de los locales de cada acción.
Esto es útil para definir un forward global de error, por ejemplo, y saber que todas las acciones podrán hacer
uso de él sin necesidad de tener que definirlo en cada una de las acciones.
Veamos un ejemplo (dentro del archivo: Configuracion/WEB-INF/struts-config.xml)
"
<!-- Ejemplo de Elemento global-forwards -->
	<global-forwards>
		<forward 
			name="mal" 
			path="/jsp/form_mal.jsp" />
	</global-forwards>
"

action-mappings
En este elemento definimos las acciones de la aplicación. Si bien hemos visto varios ejemplos en lo que 
va del libro, vamos a ver ahora una posible configuración muy útil. Al definir los action-mappings podemos
usar comodines para combinar mapeos similares y reducir el número de elementos action-mapping.
Veamos primero un ejemplo y luego la explicación (dentro del archivo: Configuracion/WEB-INF/struts-config.xml):
"
<!-- Ejemplo de uso de comodines en la definición de acciones -->
		<action
			path="/ver*" 
			type="paquete.Ver{1}Action" 
			name="{1}Form" 
			scope="request" 
			validate="false" >
			<forward name="ok" path="/jsp/{1}.jsp" />
		</action>
"

En este caso, esta meta acción se define con el comodín * que permite manejar direcciones como verSaldo, verCuentas, etc. El contenido que se asocie al comodín será reemplazado luego en las variables {1}, {2}, etc.
Como ejemplo, si ingresamos a la dirección URL: http://localhost:8080/Struts/verSaldo.do esta acción se comportará como si hubiese sido definida de la siguiente forma:
(dentro del archivo: Configuracion/WEB-INF/struts-config.xml)
"
<action
	path="/verSaldo"
	type="paquete.VerSaldoAction"
	name="SaldoForm"
	scope="request"
	validate="false" >
	<forward name="ok" path="/jsp/Saldo.jsp" />
</action>
"

Otro ejemplo más complejo, incluyendo varios comodines:
(dentro del archivo: Configuracion/WEB-INF/struts-config.xml)
"
<!-- Otro ejemplo más complejo, incluyendo varios comodines en la definición de acciones -->
		<action
			path="/*/mostrar*" 
			type="{1}.Mostrar{2}Action" 
			name="{2}Form" 
			scope="request" 
			validate="false" >
			<forward name="ok" path="/jsp/{1}/{2}.jsp" />
		</action>
"

En este ejemplo, el contenido que aparece al primer * reemplazará luego a la variable {1}
y el contenido que aparece al segundo * reemplazará a {2}. Si ingresamos en la URL 
http://localhost:8080/Struts/finanzas/mostrarBolsa.do esta acción se comportará como si hubiese sido definida:
(dentro del archivo: Configuracion/WEB-INF/struts-config.xml)
"
<action
	path="/finanzas/mostrarBolsa"
	type="finanzas.MostrarBolsaAction"
	name="BolsaForm"
	scope="request"
	validate="false" >
	<forward name="ok" path="/jsp/finanzas/Bolsa.jsp" />
</action>
"

Recordemos que la porción /Struts de la dirección URL se refiere al contexto dentro del Apache Tomcat,
y no es tomada en cuenta por Struts.

Notemos en este segundo ejemplo que el comodín * no aparea cadenas de texto que incluyan el caracter /.
Si queremos que incluya también los caracteres /, debemos usar el comodín **.

Los atributos type, name, roles, parameter, attribute, forward, include e input del elemento action aceptan comodines.
El atributo path del elemento forward también acepta comodines.

controller
Con este elemento podemos configurar a medida el controlador, o sea, el ActionServlet que recibe los pedidos
y se encarga de manejarlos y redirigirlos a las acciones correspondientes. Muchos atributos de la configuración se refieren al envío de archivos de parte del usuario, veámoslos:

bufferSize: tamaño del buffer de entrada que maneja la subida de archivos.

maxFileSize: tamaño máximo aceptable para los archivos que se suban. Expresado en bytes, si le agregamos el 
sufijo K, M o G expresa kilobytes, megabytes o gigabytes respectivamente. Por defecto: 250M.

memFileSize: tamaño máximo que será guardado en memoria cuando se está subiendo un archivo. 
Una vez alcanzado este valor, se procederá a guardar el contenido (por lo general en el disco). 
Expresado en bytes, también podemos agregar el sufijo K, M o G. Por defecto: 256K.

tempDir: carpeta temporaria donde se guardará el archivo que se está subiendo.

multipartClass: nombre de la clase que manejará los pedidos que incluyan archivos (pedidos multipart).
Por defecto: org.apache.struts.upload.CommonsMultipartRequestHandler.

contentType: tipo de contenido MIME (opcionalmente también el mapa de caracteres) que se devolverá
por defecto en cada respuesta. Este valor puede ser redefinido particularmente por cada acción o página JSP.
Por defecto, text/html.

forwardPattern: patrón que especifica cómo transformar el valor del atributo path del elemento forward 
a una dirección web (relativa al contexto). Este patrón puede contener los siguientes valores:
	$M el nombre del módulo actual 
	$P el atributo path del elemento forward
	$$ se traduce al signo $, usado como secuencia de escape.
Este atributo rara vez necesita ser modificado. Por defecto, $M$P.

pagePattern: patrón que especifica cómo transformar el valor del atributo page de los tags que lo usen 
(por ejemplo, html:link). Su formato es el mismo que en forwardPattern. Por defecto: $M$P.

inputForward: si es true, el valor del atributo input de los elementos action será tomado como referencia 
a un elemento forward, local o global. Si es false (por defecto), dicho valor será tomado como una dirección web relativa al módulo.
Veamos el siguiente ejemplo:
(dentro del archivo: Configuracion/WEB-INF/struts-config.xml)
"
<!-- Ejemplo de acción que muestra el atributo inputForward -->
		<action 
			path="/topoto"
			type="paquete.TopotoAction"
			scope="request"
			validate="true"
			input="error">
			<forward name="ok" path="/jsp/topoto.jsp" />
			<forward name="error" path="/jsp/error.jsp" />
		</action>
"
Si configuramos el elemento inputForward con valor true, al ocurrir un error de validación, 
el atributo input está indicando el nombre de un elemento forward. En este caso, lo encontrará localmente
y entonces redirigirá la acción a la página /jsp/error.jsp.

locale: si es true (por defecto lo es), se guardará en la sesión del usuario un objeto de tipo Locale.

nocache: si es true, se agregará en cada respuesta encabezados para impedir que el navegador del usuario guarde
páginas en su caché. Por defecto es false.

processorClass: nombre de la clase que realiza la lógica de procesamiento de cada pedido. Esta clase es 
muy importante y luego veremos ejemplos de cómo extender la clase que brinda Struts para agregar funcionalidad.
Por defecto es: org.apache.struts.action.RequestProcessor.


message-resources
Este elemento lo usamos para agregar a nuestra aplicación archivos de recursos (como objetos de tipo MessageResources). Podemos definir cualquier número de message-resources.
Veamos sus atributos:

factory: nombre de la subclase de org.apache.struts.util.MessageResourcesFactory que usaremos como fábrica
del objeto MessageResources. Por defecto: org.apache.struts.util.PropertyMessageResourcesFactory, que asume 
nuestro archivo de recursos es un archivo de propiedades (o sea, de la forma clave=valor).

key: nombre del atributo donde el objeto será guardado en el ServletContext. Por defecto, la constante definida
por org.apache.struts.Globals.MESSAGES_KEY

null: si es true (por defecto lo es), al buscar una clave en el archivo de recursos que no exista,
nos devolverá null. Si no, al buscar una clave inexistente, obtendremos un error.

parameter: parámetro que pasaremos al objeto fábrica para crear el objeto. En el caso de PropertyMessageResourcesFactory, el nombre del archivo de recursos (pero sin la extensión ".properties").


plug-in
Este elemento enchufa una clase en la aplicación. Un plug-in es notificado al iniciarse la aplicación y 
al ser terminada. Típicamente enchufamos un plug-in para especificar configuración extra.
El ValidatorPlugin, que hemos visto en el Capítulo 8, hace exactamente esto.
Veamos cómo configurar un plug-in y luego haremos un ejemplo.
El elemento plug-in sólo necesita de un atributo:

className: nombre de la clase del plug-in.

Al ser plug-in un elemento genérico, el resto de la configuración se brinda en base a elementos set-property 
que acceden a las propiedades de la clase. Esta clase debe implementar a org.apache.struts.action.Plugin y 
proveer un constructor vacío.

Veamos un ejemplo. Definiremos un plug-in que servirá para informar cuánto tiempo estuvo la aplicación corriendo. Primero agregamos el elemento plug-in en el archivo: Configuracion/WEB-INF/struts-config.xml
"
<!-- Ejemplo de Plugin de cronómetro -->
	<plug-in class-name="CronometroPlugin">
		<set-property property="formato" value="mm:ss"/>
	</plug-in>
"
Establecemos el valor de una propiedad, formato. Nuestra clase debe proveer métodos accesores para dicha 
propiedad. Veamos la implementación de la clase Java: Configuracion/src/CronometroPlugin.java
"
//package Configuracion.src;

import java.text.SimpleDateFormat;
import java.util.Date;
import java.util.Timezone;
import javax.servlet.ServletException;

import org.apache.struts.action.ActionServlet;
import org.apache.struts.action.PlugIn;
import org.apache.struts.config.ModuleConfig;

public class CronometroPlugin implements org.apache.struts.action.PlugIn {

    private ActionServlet servlet;
    private long inicio;

    // Asignamos un valor por defecto al formato del cronómetro
    private String formato = "dd HH:mm:ss";

    // Método accesor para el formato del cronómetro
    public String getFormato() {
        return formato;
    }

    // Metodo mutador para el formato del cronómetro
    public void setFormato(String formato) {
        this.formato = formato;
    }

    // Método llamado al iniciarse la aplicación
    public void init(ActionServlet servlet, ModuleConfig config) throws ServletException {

        // Guardamos el servlet
        this.servlet = servlet;

        // Guardamos la fecha en que inició la aplicación
        this.inicio = System.currentTimeMillis();
    }

    // Método llamado al finalizarse la aplicación
    public void destroy() {

        // Calculamos el tiempo transcurrido desde el inicio de la aplicación
        long tiempo = System.currentTimeMillis() - this.inicio;

        SimpleDateFormat dateFormat = new SimpleDateFormat(this.formato);
        dateFormat.setTimeZone(TimeZone.getTimeZone("GMT"));

        // Lo imprimimos
        servlet.log("Tiempo transcurrido desde el inicio de la aplicación: " + dateFormat.format(new Date(tiempo)));
    }
}
"

Módulos
En aplicaciones muy grandes, muchos desarrolladores a la vez estarán continuamente modificando el archivo struts-config.xml. Con el uso de controladores de versión, las chances de sobreescribir cambios que haya hecho
otro desarrollador son mínimas. Sin embargo, hay alternativas mas eficaces.

La primera alternativa consiste en definir varios archivos de configuración. Esto se hace simplemente especificando varios archivos, separados por comas, como parámetro de inicialización al ActionServlet 
dentro del archivo Configuracion/WEB-INF/web.xml Por ejemplo:
"
<!-- Primer alternativa: muchos archivos de configuración -->
<servlet>
    <servlet-name>action</servlet-name>
    <servlet-class>org.apache.struts.action.ActionServlet</servlet-class>
    <init-param>
        <param-name>config</param-name>
        <param-value>/WEB-INF/struts-config.xml, /WEB-INF/struts-config-otro.xml, /WEB-INF/struts-config-base.xml</param-value>
    </init-param>
    <load-on-startup>2</load-on-startup>
</servlet>
"

Otra alternativa es partir la aplicación en módulos. Cada módulo es una subaplicación con su propio 
archivo de configuración. Esta alternativa es preferible a solamente especificar varios archivos de 
configuración. Cada módulo es independiente del resto. Si usamos muchos archivos de configuración en un
sólo módulo, algunas aplicaciones podrían tener el mismo nombre y pisarse. Separando en módulos, podemos
tener equipos de desarrollo trabajando por separado sin necesidad de preocuparse por la configuración global.

Al definir módulos, debemos especificar un módulo por defecto (que es lo que veníamos haciendo hasta ahora)
y cero o más módulos. La configuración de módulos se realiza en el archivo Configuracion/WEB-INF/web.xml 
definiendo parámetros de inicialización de la forma config/[modulo]. Por ejemplo:
"
<!-- 
Segunda alternativa: se separa la aplicación en módulos, cada uno con su propio archivo de configuración -->
<servlet>
    <servlet-name>action</servlet-name>
    <servlet-class>org.apache.struts.action.ActionServlet</servlet-class>
    <!-- R1 módulo por defecto -->
    <init-param>
        <param-name>config</param-name>
        <param-value>/WEB-INF/struts-config.xml</param-value>
    </init-param>
    <!-- El módulo de nombre usuarios -->
    <init-param>
        <param-name>config/usuarios</param-name>
        <param-value>/WEB-INF/struts-config-usuarios.xml</param-value>
    </init-param>
    <load-on-startup>2</load-on-startup>
</servlet>
"

Esto define el módulo por defecto y otro módulo de nombre usuarios con su respectivo archivo de configuración.
El nombre del módulo no es meramente una etiqueta, sinó que define las direcciones web asociadas al módulo.
Si nuestro contexto es Struts (como los ejemplos hasta ahora) la dirección http://localhost:8080/Struts/usuarios/accion.do pertencerá al módulo usuarios; si tal acción no existiera en el archivo de configuración del módulo, obtendremos una excepción por más que sí exista en el del módulo por defecto.

Cambiar de módulo
Si tenemos módulos en nuestra aplicación, seguramente querremos poder acceder a todos desde la navegación.
Como por lo general las direcciones son relativas al contexto y al módulo, necesitamos una forma de poder cambiar de módulo fácilmente. La clase SwitchAction cambia el módulo en base a parámetros que haya en el pedido. El valor del parámetro page indica la dirección relativa al módulo adonde dirigir la acción luego 
del cambio de módulo, y con el parámetro prefix especificamos el nombre del módulo al que queremos cambiar.

Si prefix es una cadena vacía (no confundir con que no exista el parámetro), cambiaremos al módulo por defecto.

Si definimos una acción genérica de tipo SwitchAction:

<action path="/cambiarModulo" type="org.apache.struts.action.SwitchAction" />

podemos invocarla accediendo a, por ejemplo:

http://localhost:8080/Struts/cambiarModulo.do?prefix=/usuarios&page=/index.do 

En este caso estamos cambiando al módulo usuarios y queremos acceder a la página index.do de dicho módulo.

Otra forma de cambiar de módulo es usando el atributo module del elemento forward, sea éste local o global.
Este atributo simplemente indica el módulo adonde dirigir la acción, a ser usado en conjunto con los demás atributos. Veamos un ejemplo con dos forwards, uno local y otro global, que usen este atributo:
(Dentro del archivo: Configuracion/WEB-INF/struts-config.xml)
"<!-- Forward global que cambia al módulo usuarios -->
	<global-forwards>
		<forward 
			name="aModuloUsuarios"
			module="usuarios"
			path="/index.do" />
	</global-forwards>

	<action-mappings>
	<!-- Forward local ejemplo de módulos -->
		<action
			path="/ingresoDatos"
			scope="request"
			name="datosForm"
			type="IngresoDatosAction" />

			<!-- Forward local que envía la acción a la página datos.jsp del módulo usuarios -->
			<forward name="ok" module="usuarios" path="datos.jsp" />
		</action>
	</action-mappings>
"

Por último, varios tags del paquete html tienen el atributo module, que indica el módulo que se utilizará.
Por ejemplo, en el siguiente ejemplo se crea un enlace hacia otro módulo:

<html:link module="/usuarios" path="verDatos.do" />

Registro de mensajes
Si nuestra aplicación necesita escribir mensajes, sea para auditoría de acciones realizadas, para depuración,
etc., podemos usar el proyecto Commons Logging que ya viene incluído en la distribución de Struts. Con este 
proyecto de Apache, muy sencillo de utilizar, podemos controlar los mensajes que son escritos por la aplicación
de manera muy sencilla.

La idea es que cada mensaje tiene un nivel de prioridad o importancia, y podemos configurar qué nivel de importancia queremos mostrar.

La configuración del logger (la implementación que queramos usar) tiene muchas posibilidades según la que 
elijamos. Vamos a ver un pequeño ejemplo usando SimpleLog, el logger básico de Apache Commons Logging.

Necesitamos crear archivos. Uno para decirle a Commons Logging qué implementación vamos a usar, y uno 
propio para configurar tal implementación (de ahora en más, logger).

Ambos archivos deben estar en el CLASSPATH. Una buena ubicación es la carpeta Configuracion/WEB-INF/src.
Veamos cómo configurar Commons Logging para usar SimpleLog. 
En el archivo Configuracion/WEB-INF/src/commons-logging.properties 
definimos la implementación a usar de la siguiente forma:
"
org.apache.commons.logging.Log=org.apache.commons.logging.impl.SimpleLog
"
Esa clave define el logger que se utilizará, 
y de valor le proporcionamos el nombre completo de la clase SimpleLog.

Ahora debemos configurar el logger SimpleLog. Esto lo haremos también mediante otro archivo. 
Las propiedades que acepta SimpleLog son:

org.apache.commons.logging.simplelog.defaultlog: nivel de registro para todos los loggers. Por defecto, info.

org.apache.commons.logging.simplelog.log.[xyz]: nivel de registro para el logger de nombre [xyz].

org.apache.commons.logging.simplelog.showlogname: si es true muestra el nombre del Log junto con el mensaje.
Por defecto es false.

org.apache.commons.logging.simplelog.showShortLogname: si es true (por defecto) muestra el último componente 
del nombre del Log junto con el mensaje, o sea, una representación corta del nombre.

org.apache.commons.logging.simplelog.showdatetime: si es true muestra la hora y fecha del mensaje. Por defecto es false.

org.apache.commons.logging.simplelog.dateTimeFormat: el patrón que se usará para formatear la fecha.
Por defecto, yyyy/MM/dd HH:mm:ss:SSS zzz.

Hay 6 niveles de registro, en orden creciente de importancia o prioridad:
trace, debug, info, warn, error, fatal.

Si el nivel del registro por defecto es, por ejemplo info, y nuestra clase registra mensajes de nivel debug,
éstos no serán mostrados porque son de un nivel menor al que queremos ver.

Sólo los mensajes de la importancia del nivel de registro o mayor se mostrarán.

Veamos un ejemplo concreto de configuración de SimpleLog:

El otro Archivo de Configuración del logger: Configuracion/WEB-INF/src/simplelog.properties
 "
org.apache.commons.logging.simpleLog.defaultlog=trace
 "
Con esta configuración estamos especificando que queremos ver todos los mensajes de nivel trace y, por 
consiguiente, todos los mensajes para todas las clases.

Con esta configuración, si iniciamos Tomcat veremos una cantidad gigantesca de mensajes, arrojados por Struts,
Tomcat, etc. al iniciarse.

Lo que queremos hacer entonces es mantener un nivel de registro para nuestras clases, y restringir el nivel 
de registro para el resto de las clases. Esto lo podemos hacer fácilmente de la siguiente forma:
(Archivo: Configuracion/WEB-INF/src/simplelog.properties)
"
org.apache.commons.logging.simpleLog.defaultlog=warn
org.apache.commons.logging.simplelog.log.configuracion=trace
"
Estamos definiendo que por defecto sólo se registren los mensajes de nivel warn o más, y que para el 
logger de nombre configuracion, se registren todos los mensajes.

Los logger tienen un nivel jerárquico. Si creamos los loggers basados en el nombre de las clases, podremos
agruparlos en base a paquetes y establecer el nivel de registro jerárquicamente, por ejemplo:
(Archivo: Configuracion/WEB-INF/src/simplelog.properties)
"
org.apache.commons.logging.simpleLog.defaultlog=warn
org.apache.commons.logging.simplelog.log.org.apache=info
org.apache.commons.logging.simplelog.log.configuracion=trace
"
Define que para todos los loggers de nombre org.apache el nivel de registro sea info. 
Esto hace que efectivamente todas las clases de los proyectos de Apache tengan ese nivel de registro.

Desde las aplicaciones, es muy sencillo acceder al logger, veamos el siguiente código Java:
Archivo: Configuracion/WEB-INF/src/ejemploLogger.java 
"
//package Configuracion.WEB-INF.src;
package configuracion;

import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import org.apache.commons.logging.Log;
import org.apache.commons.logging.LogFactory;
import org.apache.struts.action.Action;
import org.apache.struts.action.ActionForm;

public class EjemploLogger extends Action 
{

    private static final Log log = LogFactory.getLog(EjemploLogger.class);

    public ActionForward execute(ActionMapping map, ActionForm form,
            HttpServletRequest req, HttpServletResponse res)
            throws Exception 
    {
        log.trace("Inicia la acción EjemploLogger");
        
        try
        {
            int edad = Integer.parseInt(req.getParameter("edad"));
        }
        catch (NumberFormatException nfe)
        {
            log.warn("Edad inválida", nfe);
        }

        return map.findForward("ok");
    }
    
}
"
Es conveniente crear el logger con el nombre de la clase que lo invocará, como por ejemplo LoggerAction, 
para poder aprovechar su estructura jerárquica.

La jerarquía de los loggers es una organización a nivel sintáctico solamente. El logger no accede a las clases 
para comprobar su existencia. En el ejemplo, el nombre del logger será configuracion.EjemploLogger; si no existe tal configuración buscará para "configuración", y si tampoco existe, usará la configuración por defecto.

Extender la funcionalidad de Struts
Struts es un proyecto de código abierto. Cualquiera puede descargar gratuitamente el código fuente de su página
web y modificarlo para ajustarlo a las necesidades propias. Si bien esto es perfectamente factible, recordemos que Struts es un proyecto en constante desarrollo por una comunidad internacional, y nuevas versiones son publicadas periódicamente. Si modificamos directamente el código de Struts, probablemente tengamos complicaciones si queremos actualizar a una nueva versión, ya que perderemos nuestro código al actualizar Struts y debamos volver a hacer los cambios cada vez, incluso con el riesgo de que nuestros cambios no sean compatibles con nuevas versiones.

Pero Struts también está hecho en Java, y su orientación a objetos hace que sea fácil extender su funcionalidad. Más aun, hemos visto que muchas veces la configuración nos permite definir qué clase queremos 
usar para realizar determinadas tareas. Por defecto, estas clases son las que provee Struts, pero nada nos impide crear nuestras propias clases y usarlas.

Hemos visto que podemos definir plug-ins para ejecutar código al iniciarse y finalizarse la aplicación. 
Esta es una forma de extender la funcionalidad de Struts, veamos de qué otra forma podemos extenderlo.

RequestProcessor
Esta clase es la encargada de realizar toda la lógica de cada pedido. Al recibir un pedido, la clase ActionServlet invoca a una instancia de esta clase pasándole los objetos HttpServletRequest y HttpServletResponse. 

El método principal de RequestProcessor llama a varios métodos más pequeños. Esta modularidad nos facilita la 
tarea de extender la clase, ya que podemos redefinir solamente los métodos que necesitemos sin necesidad de 
modificar el método principal.

Un esqueleto del código del método process() el (el método principal llamado por ActionServlet) 
de la clase RequestProcessor vemos a continuación:
"
/* Se obtiene la dirección para obtener la acción a ejecutar */
String path = processPath(request, response);

/* Obtiene el Locale y lo guarda en sesión (si la configuración así lo indica) */
processLocale(request, response);

/* Obtiene el tipo de contenido a devolver de la configuración */
processContent(request, response);

/* Agrega encabezados para evitar la caché de los navegadores (si la configuración así lo indica) */
processNoCache(request, response);

/* Llamado a un método genérico, ideal para redefinir en una subclase */
if(!processPreprocess(request, response))
{
	return;
}

/* Quita de la sesión los ActionMessages que fueron accedidos al menos una vez */
this.processCachedMessages(request, response);

/* Obtiene el ActionMapping que procesará el pedido */
ActionMapping mapping = processMapping(request, response, path);
if(mapping == null)
{
	return;
}

/* Verifica que el usuario tenga los roles de seguridad necesarios para acceder al recurso */
if(!processRoles(request, response, mapping))
{
	return;
}

/* Obtiene el form asociado al mapping */
ActionForm form = processActionForm(request, response, mapping);

/* Llena los atributos del form, en base al pedido */
processPopulate(request, response, form, mapping);

/* Llama al método validate del form, y en caso de haber errores redirige a la página de error */
if(!processValidate(request, response, form, mapping))
{
	return;
}

/* Si hay un atributo forward en el mapping, se lo invoca */
if(!processForward(request, response, mapping))
{
	return;
}

/* Si hay un atributo include en el mapping, se lo invoca */
if(!processInclude(request, response, mapping))
{
	return;
}

/* Crea u obtiene la clase Action que va a procesar este pedido */
Action action = porcessActionCreate(request, response, mapping);
if(action == null)
{
	return;
}

/* Invoca la instancia de la acción y obtiene el ActionForward a donde redirigir el pedido */
ActionForward forward = processActionPerform(request, response, action, form, mapping);

/* Redirige el pedido a donde especifique el objeto ActionForward */
processForwardConfig(request, response, forward);
"

Algunos de estos métodos interactúan con muchos objetos del framework, y debemos ser muy cuidadosos
si queremos modificarlos o extenderlos.

Esta clase es básicamente todo Struts desde adentro. Aquí se comprende en concreto el funcionamiento de
todo lo que vimos en los capítulos anteriores.

El método processPreprocess() es un método creado específicamente para extensibilidad. Este método es 
llamado, pero su definición en la clase RequestProcessor es vacía. Fácilmente podemos extender la clase 
redefiniendo este método. El método debe devolver un valor booleano. Si devolvemos true, indicamos que el 
procesamiento debe continuar, caso contrario estamos indicando que ya hemos procesado el pedido.

Vamos a un ejemplo. Queremos que la aplicación esté disponible a los usuarios sólo en determinado horario. 
Extendemos la clase RequestProcessor: 
Archivo: Configuracion/WEB-INF/src/RequestProcessorHorario.java 
"
//package Configuracion.WEB-INF.src;
package configuracion;

import java.io.IOException;
import java.util.Calendar;
import java.util.GregorianCalendar;
import java.util.TimeZone;

import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;

import org.apache.commons.logging.Log;
import org.apache.commons.logging.LogFactory;
import org.apache.struts.action.RequestProcessor;

public class RequestProcessorHorario extends RequestProcessor {
    protected static Log log = LogFactory.getLog(RequestProcessorHorario.class);

    @Override
    /**
     * Verificamos estar dentro de un rango horario. Si lo estamos, dejamos que
     * RequestProcessor
     * siga con el procesamiento del pedido, sino, reenviamos a una página que
     * muestre un mensaje.
     * 
     * @param request
     * @param response
     * @return
     * @throws IOException
     */
    protected boolean processPreprocess(HttpServletRequest request, HttpServletResponse response) throws IOException {
        GregorianCalendar gc = new GregorianCalendar(TimeZone.getTimeZone("GMT"));

        int hora = gc.get(Calendar.HOUR_OF_DAY);

        if (hora < 9 || hora > 15) {
            try {
                // Reenviamos a una página que muestre un mensaje
                response.sendRedirect(response.encodeRedirectURL("fueraDeHorario.html"));
            } catch (IOException ioe) {
                // Registramos el error
                log.error("Error al redirigir por fuera de horario", ioe);
            }

            // Cortamos el procesamiento
            return false;
        } else {
            // Que siga el procesamiento normal
            return true;
        }
    }
}
"

Recordemos que para usar una clase distinta a RequestProcessor debemos configurarlo en el elemento 
controller del archivo Configuracion/WEB-INF/struts-config.xml 

ActionServlet
Esta clase es la que recibe todos los pedidos. La lógica de manejo de pedidos vimos que la delega a RequestProcessor. La funcionalidad de ActionServlet está en los métodos de inicialización, donde se leen 
los archivos de configuración, se interpretan, etc.

En general, no vamos a necesitar extender a ActionServlet a no ser que queramos interactuar directamente 
con elementos propios de Struts en profundidad. Si necesitamos agregar funcionalidad al iniciar la aplicación,
podemos usar un plug-in y para funcionalida en la ejecución, extendemos a RequestProcessor. Si nos encontramos
con una situación en que necesitamos agregar la funcionalidad sí o sí en ActionServlet, podemos extender la 
clase e invocarla directamente desde el archivo Configuracion/WEB-INF/web.xml; recordemos que ActionServlet no deja de ser un servlet.

Supongamos que queremos una aplicación que sólo responda a métodos HTTP GET. Podríamos hacer esto fácilmente 
con una subclase de RequestProcessor, usando el método getMethod() de la clase HttpServletRequest y filtrando 
todo pedido que no sea del tipo que queremos. Pero supongamos que por alguna razón no podemos resolver el problema con ese enfoque. Desarrollemos una subclase de ActionServlet. ActionServlet define los métodos 
doGet() y doPost(). Los otros métodos correspondientes a los otros tipos de pedido (doDelete(), doPut(), etc.) 
no son redefinidos y la implementación por defecto no devuelve nada. Debemos entonces redefinir doPost() para que este método no sea atendido por el servidor:
Archivo: Configuracion/WEB-INF/src/GetActionServlet.java 
"
//package Configuracion.WEB-INF.src;
package configuracion;

import java.io.IOException;

//import javax.servlet.ServletException;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;

import org.apache.struts.action.ActionServlet;

/**
 * Subclase de ActionServlet que sólo atiende pedidos HTTP GET
 */

public class GetActionServlet extends ActionServlet {
    @Override
    /**
     * Sobreescribimos este método para que la superclase no lo atienda
     * 
     * @param request
     * @param response
     * @throws IOException
     * @throws ServletException
     */
    public void doPost(HttpServletRequest request, HttpServletResponse response) throws IOException, ServletException {
        // No hacemos nada, sólo atendemos GET
    }
}
"

Luego en el archivo Configuracion/WEB-INF/web.xml definimos que queremos nuestra versión GetActionServlet:
"
<!-- Ejemplo de configuración para usar GetActionServlet -->
<servlet>
    <servlet-name>action</servlet-name>
    <servlet-class>configuracion.GetActionServlet</servlet-class>
    <init-param>
        <param-name>config</param-name>
        <param-value>/WEB-INF/struts-config.xml</param-value>
    </init-param>
    <load-on-startup>2</load-on-startup>
</servlet>
"
